が必要か\n\nReact には Flutter の InheritedWidget や Provider に相当する標準的な状態管理の仕組みが（当初は）ありませんでした。そのため、コミュニティが Redux という状態管理ライブラリを開発しました。\n\n```\nFlutter の場合:\nWidget Tree の上位で Provider/Riverpod を使って状態を提供\n  ↓\n子孫の Widget で Consumer/useProvider を使って状態を取得\n\nReact + Redux の場合:\nComponent Tree の上位で Redux Store を Provider で提供\n  ↓\n子孫の Component で useSelector を使って状態を取得\n```\n\n---\n\n## Flutter との概念対応表\n\n| Flutter (Riverpod) | React + Redux | 説明 |\n|-------------------|---------------|------|\n| `ProviderContainer` / `ProviderScope` | `<Provider store={store}>` | 状態管理の範囲を定義 |\n| `StateNotifierProvider` | `createSlice` (Redux Toolkit) | 状態とその更新ロジックを定義 |\n| `StateNotifier` | Reducer + Actions | 状態の変更方法を定義 |\n| `ref.read(provider)` | `useDispatch()` | 状態を変更するためのアクション発行 |\n| `ref.watch(provider)` | `useSelector(state => ...)` | 状態を購読して取得 |\n| `provider.state` | Redux Store の `state` | 実際の状態の値 |\n| `.notifier.update()` | `dispatch(actions.setXxx())` | 状態の更新 |\n| `AsyncValue` | 独自実装が必要 | 非同期状態の管理（Redux には標準でない） |\n\n---\n\n## Redux の基本概念\n\nRedux は **3 つの原則** に基づいて設計されています。\n\n### 原則 1: Single Source of Truth（信頼できる唯一の情報源）\n\n**全ての状態は、1 つの Store オブジェクトに保存される。**\n\n```dart\n// Flutter (Riverpod)\nfinal userProvider = StateNotifierProvider(...);\nfinal teamProvider = StateNotifierProvider(...);\nfinal settingsProvider = StateNotifierProvider(...);\n// → 複数の Provider が独立して存在\n\n// React (Redux)\n// → 全ての状態が 1 つの Store に集約される\n{\n  user: { ... },\n  team: { ... },\n  settings: { ... }\n}\n```\n\n### 原則 2: State is Read-Only（状態は読み取り専用）\n\n**状態を変更する唯一の方法は、Action を発行すること。**\n\n```dart\n// Flutter (Riverpod)\nref.read(counterProvider.notifier).increment();\n// → notifier を通じて直接メソッドを呼ぶ\n\n// React (Redux)\ndispatch(counterActions.increment());\n// → Action を dispatch して間接的に変更\n```\n\nこの仕組みにより、**いつ、どこで、なぜ状態が変わったか**を追跡できます（Time Travel Debugging）。\n\n### 原則 3: Changes are Made with Pure Functions（変更は純粋関数で行う）\n\n**状態を変更する Reducer は、副作用のない純粋関数でなければならない。**\n\n```dart\n// Flutter (Riverpod)\nclass CounterNotifier extends StateNotifier<int> {\n  void increment() {\n    state = state + 1; // 直接代入で更新\n  }\n}\n\n// React (Redux - 古い書き方)\nfunction counterReducer(state = 0, action) {\n  switch (action.type) {\n    case 'INCREMENT':\n      return state + 1; // 新しい値を return（不変性）\n    default:\n      return state;\n  }\n}\n```\n\n**重要**: Redux では状態を直接変更せず、**常に新しい状態オブジェクトを返す**必要があります（Immutability）。\n\n---\n\n## Redux Toolkit - モダンな Redux\n\n**Redux Toolkit** は、従来の Redux のボイラープレート（定型コード）を削減するための公式ツールです。\n\n### 従来の Redux の問題点\n\n```javascript\n// 1. Action Type の定義\nconst INCREMENT = 'counter/INCREMENT';\n\n// 2. Action Creator の定義\nconst increment = () => ({ type: INCREMENT });\n\n// 3. Reducer の定義\nfunction counterReducer(state = 0, action) {\n  switch (action.type) {\n    case INCREMENT:\n      return state + 1;\n    default:\n      return state;\n  }\n}\n\n// 4. が必要か\n\nReact には Flutter の InheritedWidget や Provider に相当する標準的な状態管理の仕組みが（当初は）ありませんでした。そのため、コミュニティが Redux という状態管理ライブラリを開発しました。\n\n```\nFlutter の場合:\nWidget Tree の上位で Provider/Riverpod を使って状態を提供\n Flutter (Riverpod) | React + Redux | 説明 |\n|-------------------|---------------|------|\n| `ProviderContainer` / `ProviderScope` | `<Provider store={store}>` | 状態管理の範囲を定義 |\n| `StateNotifierProvider` | `createSlice` (Redux Toolkit) | 状態とその更新ロジックを定義 |\n| `StateNotifier` | Reducer + Actions | 状態の変更方法を定義 |\n| `ref.read(provider)` | `useDispatch()` | 状態を変更するためのアクション発行 |\n| `ref.watch(provider)` | 非同期状態の管理（Redux には標準でない） |\n\n---\n\n## Redux の基本概念\n\nRedux は **3 つの原則** に基づいて設計されています。\n\n### 原則 1: Single Source of Truth（信頼できる唯一の情報源）\n\n**全ての状態は、1 つの Store オブジェクトに保存される。**\n\n```dart\n// Flutter (Riverpod)\nfinal userProvider = StateNotifierProvider(...);\nfinal teamProvider = StateNotifierProvider(...);\nfinal settingsProvider = StateNotifierProvider(...);\n// → 複数の Provider が独立して存在\n\n// React (Redux)\n// → 全ての状態が 1 つの Store に集約される\n{\n  user: { ... },\n  team: { ... },\n  settings: { ... }\n}\n```\n\n### 原則 2: State is Read-Only（状態は読み取り専用）\n\n**状態を変更する唯一の方法は、Action を発行すること。**\n\n```dart\n// Flutter (Riverpod)\nref.read(counterProvider.notifier).increment();\n// → notifier を通じて直接メソッドを呼ぶ\n\n// React (Redux)\ndispatch(counterActions.increment());\n// → Action を dispatch して間接的に変更\n```\n\nこの仕組みにより、**いつ、どこで、なぜ状態が変わったか**を追跡できます（Time Travel Debugging）。\n\n### 原則 3: Changes are Made with Pure Functions（変更は純粋関数で行う）\n\n**状態を変更する Reducer は、副作用のない純粋関数でなければならない。**\n\n```dart\n// Flutter (Riverpod)\nclass CounterNotifier extends StateNotifier<int> {\n  void increment() {\n    state = state + 1; // 直接代入で更新\n  }\n}\n\n// React (Redux - 古い書き方)\nfunction counterReducer(state = 0, action) {\n  switch (action.type) {\n    case 'INCREMENT':\n      return state + 1; // 新しい値を return（不変性）\n    default:\n      return state;\n  }\n}\n```\n\n**重要**: Redux では状態を直接変更せず、**常に新しい状態オブジェクトを返す**必要があります（Immutability）。\n\n---\n\n## Redux Toolkit - モダンな Redux\n\n**Redux Toolkit** は、従来の Redux のボイラープレート（定型コード）を削減するための公式ツールです。\n\n### 従来の
